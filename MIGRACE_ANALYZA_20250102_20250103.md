# Analýza chybějících migrací 20250102 a 20250103

## 1. Kde se tyto verze berou?

### Supabase CLI tracking migrací

Supabase CLI trackuje verze migrací v databázi v tabulce:
- **`supabase_migrations.schema_migrations`** (nebo ekvivalentní)

Tato tabulka obsahuje:
- `version` (BIGINT) - timestamp verze migrace (např. 20250102000000)
- `name` (TEXT) - název migrace (název souboru bez přípony)
- `statements` (TEXT[]) - pole SQL příkazů z migrace (volitelné, záleží na verzi CLI)

### Jak CLI čte verze

1. **Lokálně**: CLI čte soubory z `supabase/migrations/*.sql` a porovnává je s remote databází
2. **Remote**: CLI se dotazuje na `supabase_migrations.schema_migrations` tabulku v remote databázi
3. **Konflikt**: Pokud jsou verze v remote DB, ale nejsou v lokálních souborech, CLI to detekuje jako nekonzistenci

### Ověření v databázi

Pro ověření spusť v Supabase SQL Editor:

```sql
-- Zkontroluj, zda existuje tabulka pro tracking migrací
SELECT table_schema, table_name 
FROM information_schema.tables 
WHERE table_name LIKE '%migration%' OR table_name LIKE '%schema%';

-- Zkontroluj verze migrací v remote DB
SELECT version, name, statements 
FROM supabase_migrations.schema_migrations 
WHERE version IN (20250102000000, 20250103000000, 20250103000001, 20250103000002)
ORDER BY version;

-- Zkontroluj všechny migrace
SELECT version, name, inserted_at 
FROM supabase_migrations.schema_migrations 
ORDER BY version;
```

3. **Byly revertnuty** - migrace byly aplikovány a pak revertnuty pomocí `supabase migration repair`

## 3. Nevznikly automaticky při db pull/diff?

### Možné příčiny automatického vytvoření

#### A) `supabase db diff`
```bash

supabase db diff -f 20250102000000_some_name
# CLI vytvoří nový migrační soubor s aktuálním timestampem
```

#### B) `supabase migration new`
```bash
# Pokud někdo udělal:
supabase migration new some_name
# CLI vytvoří soubor s aktuálním timestampem
```

#### C) `supabase db pull`
```bash
# Pokud někdo udělal:
supabase db pull
# CLI může vytvořit migrace pro změny, které jsou v remote DB, ale nejsou v lokálních migracích
```

### Proč jsou v remote jako "reverted"?

Status "reverted" v Supabase znamená:
- Migrace byla **aplikována** na databázi
- Pak byla **označena jako revertnutá** (buď manuálně, nebo automaticky)
- **SQL změny z migrace mohou být stále v databázi** (revert = pouze označení, ne rollback!)

⚠️ **DŮLEŽITÉ**: "reverted" status **NENÍ** automatický rollback SQL změn. Je to pouze metadata flag.

## 4. Co znamená status "reverted" a je bezpečné je ignorovat?

### Co je "reverted" status

V Supabase CLI kontextu:
- **"reverted"** = migrace je označena jako neplatná/zrušená v tracking tabulce
- **NENÍ to automatický rollback** - SQL změny z migrace mohou být stále v databázi
- Používá se pro označení migrací, které byly aplikovány, ale pak se zjistilo, že jsou neplatné/duplicitní

### Je bezpečné je ignorovat?

**ZÁLEŽÍ na tom, co ty migrace obsahovaly:**

1. **Pokud byly prázdné/duplicitní**: ✅ Bezpečné ignorovat
2. **Pokud obsahovaly důležité změny**: ❌ **NEBEZPEČNÉ** - změny mohou být stále v DB, ale nejsou v git historii

### Ověření, zda jsou změny v DB

```sql
-- Zkontroluj, zda existují objekty, které by mohly být z těch migrací
-- (záleží na tom, co migrace obsahovaly)

-- Příklady kontrol:
SELECT * FROM information_schema.tables 
WHERE table_schema = 'public' 
AND table_name LIKE '%test%' OR table_name LIKE '%temp%';

-- Zkontroluj funkce
SELECT routine_name, routine_definition 
FROM information_schema.routines 
WHERE routine_schema = 'public' 
AND created_at >= '2025-01-02' AND created_at < '2025-01-04';
```

## 7. Jaký je správný fix?

### Analýza možností

#### (A) Obnovit původní 4 migrační soubory z git historie

**Kdy použít:**
- ✅ Pokud migrace existují v git historii
- ✅ Pokud obsahují důležité změny, které nejsou v jiných migracích
- ✅ Pokud chceš zachovat kompletní historii

**Jak:**
```bash
# Najdi commit, kde byly migrace
git log --all --oneline -- "supabase/migrations/20250102*.sql"

# Obnov je
git checkout <commit-hash> -- supabase/migrations/20250102000000_*.sql
git checkout <commit-hash> -- supabase/migrations/20250103000000_*.sql
# atd.
```

#### (B) Vytvořit prázdné placeholdery (protože status je reverted)

**Kdy použít:**
- ✅ Pokud migrace byly prázdné/duplicitní
- ✅ Pokud změny z migrací jsou už v jiných migracích
- ✅ Pokud chceš jen srovnat historii bez duplikace změn

**Jak:**
```bash
# Vytvoř prázdné placeholdery
touch supabase/migrations/20250102000000_placeholder.sql
touch supabase/migrations/20250103000000_placeholder.sql
touch supabase/migrations/20250103000001_placeholder.sql
touch supabase/migrations/20250103000002_placeholder.sql

# Obsah každého souboru:
-- Migration placeholder
-- This migration was marked as reverted in remote database
-- No SQL changes needed as they were already applied or are duplicates
```

#### (C) `supabase migration repair`

**Kdy použít:**
- ✅ Pokud chceš synchronizovat lokální a remote stav
- ✅ Pokud CLI detekuje nekonzistenci a nabízí repair

**Jak:**
```bash
# CLI může nabídnout repair automaticky při:
supabase db push

# Nebo explicitně:
supabase migration repair
```

### **DOPORUČENÍ: Kombinace (B) + ověření**

1. **Nejprve ověř v remote DB**, co ty migrace obsahovaly:
   ```sql
   SELECT version, name, statements 
   FROM supabase_migrations.schema_migrations 
   WHERE version IN (20250102000000, 20250103000000, 20250103000001, 20250103000002);
   ```

2. **Pokud byly prázdné/duplicitní**: Vytvoř prázdné placeholdery (B)

3. **Pokud obsahovaly důležité změny**: Obnov je z git historie (A) nebo vytvoř nové migrace s těmito změnami

4. **Po fixu**: Spusť `supabase db push` a ověř, že vše funguje

## 8. Zkontroluj, zda remote historie neobsahuje duplicity nebo nekonzistence

### SQL dotaz pro kontrolu

```sql
-- Zkontroluj duplicity verzí
SELECT version, COUNT(*) as count
FROM supabase_migrations.schema_migrations
WHERE version IN (20250102000000, 20250103000000, 20250103000001, 20250103000002)
GROUP BY version
HAVING COUNT(*) > 1;

-- Zkontroluj pořadí migrací
SELECT version, name, inserted_at
FROM supabase_migrations.schema_migrations
WHERE version >= 20250101000000 AND version < 20250104000000
ORDER BY version;

-- Zkontroluj, zda nejsou mezery v sekvenci
WITH ordered AS (
  SELECT version, 
         LAG(version) OVER (ORDER BY version) as prev_version
  FROM supabase_migrations.schema_migrations
  WHERE version >= 20250101000000
)
SELECT version, prev_version, 
       version - prev_version as gap
FROM ordered
WHERE prev_version IS NOT NULL
ORDER BY version;
```

## 9. Po fixu: Jak zabránit opakování?

### Workflow pro migrace

#### 1. Kdo vytváří migrace

**Pravidlo**: Pouze jeden člověk v týmu vytváří migrace v daný den, nebo použijte branch workflow.

**Doporučení**:
- Vytvářej migrace v feature branchech
- Review migrací před merge do main
- Používej `supabase migration new <descriptive_name>` místo ručního vytváření

#### 2. Co se commituje

**VŽDY commitni:**
- ✅ Všechny migrační soubory (`supabase/migrations/*.sql`)
- ✅ Změny v `supabase/config.toml` (pokud jsou)
- ✅ Změny v seed souborech (pokud jsou)

**NIKDY necommitni:**
- ❌ `.env` soubory
- ❌ Lokální Supabase data (`supabase/.temp/`)

#### 3. Jak se dělá pull/push v týmu

**Workflow:**

```bash
# 1. Před prací: Pull nejnovější změny
git pull origin main

# 2. Vytvoř novou migraci
supabase migration new add_new_feature

# 3. Napiš SQL do migrace
# ... edit supabase/migrations/YYYYMMDDHHMMSS_add_new_feature.sql

# 4. Otestuj lokálně
supabase db reset  # Aplikuje všechny migrace
# ... test aplikace

# 5. Commit a push
git add supabase/migrations/
git commit -m "feat: add new feature migration"
git push origin feature-branch

# 6. Po merge do main: Ostatní členové týmu
git pull origin main
supabase db pull  # Synchronizuje remote změny (pokud byly aplikovány přes Dashboard)
```

#### 4. Jak se řeší revertované migrace

**Pokud se stane, že migrace je revertnutá:**

1. **Zjisti důvod**:
   ```sql
   SELECT version, name, statements, inserted_at
   FROM supabase_migrations.schema_migrations
   WHERE version = <reverted_version>;
   ```

2. **Ověř, zda změny jsou v DB**:
   - Zkontroluj, zda objekty z migrace existují
   - Pokud ano, změny jsou stále v DB (revert = pouze metadata)

3. **Rozhodni se**:
   - **Pokud změny jsou v DB a jsou správné**: Vytvoř placeholder migraci
   - **Pokud změny jsou v DB a jsou špatné**: Vytvoř rollback migraci
   - **Pokud změny nejsou v DB**: Ignoruj nebo obnov z git historie

4. **Synchronizuj**:
   ```bash
   # Pokud CLI detekuje nekonzistenci
   supabase migration repair
   
   # Nebo manuálně vytvoř placeholder
   touch supabase/migrations/YYYYMMDDHHMMSS_placeholder.sql
   ```

### Checklist před push migrace

- [ ] Migrace má popisný název
- [ ] Migrace je idempotentní (používá `IF NOT EXISTS`, `IF EXISTS`, atd.)
- [ ] Migrace je otestovaná lokálně (`supabase db reset`)
- [ ] Migrace je commitnutá do git
- [ ] Remote DB je synchronizovaná (`supabase db push` nebo přes Dashboard)
- [ ] Ostatní členové týmu jsou informováni o změnách

### Automatizace (volitelné)

**Pre-commit hook:**
```bash
#!/bin/bash
# .git/hooks/pre-commit

# Zkontroluj, zda jsou všechny migrace commitnuté
if git diff --cached --name-only | grep -q "supabase/migrations/"; then
  echo "✓ Migration files detected"
  
  # Zkontroluj syntaxi SQL (volitelné)
  for file in $(git diff --cached --name-only | grep "supabase/migrations/.*\.sql$"); do
    if ! psql -f "$file" --dry-run 2>/dev/null; then
      echo "⚠️  Warning: SQL syntax check failed for $file"
    fi
  done
fi
```
